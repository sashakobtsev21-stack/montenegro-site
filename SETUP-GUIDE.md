# SETUP-GUIDE — пошаговый запуск проекта в Claude Code (для чайников)

Цель: с нуля получить рабочее окружение, где Claude Code с твоей командой агентов (my_agents) и дизайн-скиллом (ui-ux-pro-max) строит сайт по SPEC.md. Все команды копируются как есть. Примеры путей для Windows — поправь под себя (`C:\projects\...`); на macOS/Linux — `~/projects/...`.

---

## Шаг 0. Картина мира (важно понять до команд)

Будет ТРИ отдельные вещи:

```
~/projects/gruzia-site        ← ОСНОВНОЙ проект (сайт). Его создаём. Это git-репозиторий.
my_agents                     ← инструмент №1: команда агентов для Claude Code
ui-ux-pro-max-skill           ← инструмент №2: дизайн-скилл для Claude Code
```

Ключевая мысль: **«подключить репозитории к основному проекту» ≠ копировать их код в проект.** Оба репозитория — это плагины для Claude Code. Они подключаются к самому Claude Code (через встроенный plugin marketplace) и после этого доступны в любом твоём проекте. Репозиторий сайта остаётся чистым: только код сайта, контент и спека. Не делай их git-сабмодулями и не клонируй внутрь gruzia-site — это захламит git и ничего не даст.

## Шаг 1. Подготовка машины (один раз)

1. Node.js версии 20 или новее. Проверка:
   ```bash
   node -v        # должно быть v20.x или выше
   ```
   Нет или старее — поставить с nodejs.org (LTS).
2. Git установлен и настроен:
   ```bash
   git --version
   git config --global user.name  "Имя"
   git config --global user.email "почта@как-на-github"
   ```
3. Claude Code установлен и обновлён:
   ```bash
   npm install -g @anthropic-ai/claude-code   # если ещё не стоит
   claude --version
   claude update                              # plugin marketplace требует свежей версии
   ```
4. Аккаунт GitHub есть, вход в браузере выполнен.

## Шаг 2. Создаём основной проект

```bash
mkdir -p ~/projects/gruzia-site
cd ~/projects/gruzia-site
git init
```

Положи в корень два файла из этой переписки: **SPEC.md** и **CLAUDE.md** (новые версии 2.0). Затем:

```bash
git add SPEC.md CLAUDE.md
git commit -m "spec: v2.0 — спецификация и правила проекта"
```

Создай на github.com новый **приватный** пустой репозиторий `gruzia-site` (без README — он уже есть локально... точнее, README добавим позже; главное — без автосозданных файлов) и привяжи:

```bash
git branch -M main
git remote add origin https://github.com/ТВОЙ-ЛОГИН/gruzia-site.git
git push -u origin main
```

Проверка: файлы видны на GitHub.

## Шаг 3. Подключаем my_agents (твоя команда агентов)

**Способ А — рекомендуемый: plugin marketplace (ничего не клонируем).**
Запусти Claude Code в папке проекта и выполни внутри сессии:

```
cd ~/projects/gruzia-site
claude
```
В сессии Claude Code:
```
/plugin marketplace add sashakobtsev21-stack/my_agents
/plugin install ruflo-core@ruflo
/plugin install ruflo-swarm@ruflo
```
Это базовый набор. Остальные плагины — по мере надобности, когда дойдём до соответствующих фаз:
```
/plugin install ruflo-testgen@ruflo          # фаза тестов (R3+)
/plugin install ruflo-security-audit@ruflo   # гейт безопасности (R4–R5)
/plugin install ruflo-docs@ruflo             # документация (по желанию)
```

**Способ Б — альтернатива: локальная копия агентов прямо в проект.**
Нужен, если хочешь видеть агентов файлами в `.claude/` проекта и точечно их править:
```bash
git clone https://github.com/sashakobtsev21-stack/my_agents ~/tools/my_agents
cd ~/tools/my_agents && npm install
node scripts/connect-to-project.mjs ~/projects/gruzia-site
```
Скрипт сам определит профиль проекта и скопирует ядро агентов + пак под тип проекта в `gruzia-site/.claude/`.

Начни со способа А. Не делай А и Б одновременно для одних и тех же агентов — будут дубли. Если команды `/plugin ...` ведут себя иначе, чем здесь написано — открой README репозитория my_agents: он первичен, синтаксис мог обновиться.

**Проверка:** в сессии Claude Code набери `/agents` (должны появиться роли вроде coder/reviewer/tester) и `/help` (должны появиться скиллы ruflo: swarm, witness, testgen, audit, docs). Примечание: `/new-project` и `/analyze-project` в установленный набор НЕ входят (они в неустановленных планировочных плагинах) — старт фаз ведём через скилл `swarm` + агентов `architect`/`astro-platform-engineer` (см. SPEC §19).

## Шаг 4. Подключаем ui-ux-pro-max (дизайн-скилл)

**Способ А — plugin marketplace (рекомендуемый), в той же сессии:**
```
/plugin marketplace add nextlevelbuilder/ui-ux-pro-max-skill
/plugin install ui-ux-pro-max@ui-ux-pro-max-skill
```

**Способ Б — через их CLI (кладёт скиллы файлами в проект):**
```bash
cd ~/projects/gruzia-site
npx uipro-cli init --ai claude
```

**Проверка:** скажи Claude в сессии: «Используя скилл ui-ux-pro-max, предложи дизайн-систему по брифу из SPEC.md §10». В ответе должны появиться стиль/палитра/типографика в фирменном формате скилла.

## Шаг 5. Контрольная проверка всей связки

В сессии Claude Code (в папке gruzia-site):

```
/plugin
```
— в списке установленных должны быть плагины ruflo и ui-ux-pro-max.

Затем смоук-тест понимания проекта — попроси:
> «Прочитай CLAUDE.md и SPEC.md. Перескажи в 10 строк: что строим, какие фазы R0–R6 и какой gate у фазы R1. Ничего не делай, только пересказ.»

Если пересказ совпадает со спекой — окружение готово.

## Шаг 5.5. Фаза R0.5 — аудит команды и ре-анализ спеки

Перед каркасом проверяем саму команду: хватает ли агентов и скиллов под фазы R1–R6, и пересматриваем спеку свежей командой. Три промпта подряд, между ними — твоё ревью.

**1. Gap-анализ (ничего не создаёт):**
> «Прочитай SPEC.md и CLAUDE.md. Выведи список доступных агентов и скиллов с учётом подключённых плагинов. Построй таблицу: фаза R1–R6 → ключевые задачи фазы → какой агент/скилл закрывает → вердикт: хватает / нужна донастройка / нужен новый. Для каждого "нужен новый" — ТЗ в 5 строк: имя, зона ответственности, нужные инструменты, какие разделы SPEC обязан читать. Только анализ, ничего не создавай.»

**2. Создание недостающих (после твоего «ок» по списку):**
> «Создай агентов из утверждённого списка. Проектных — в .claude/agents/ этого репозитория; универсальных — подготовь файлами для коммита в my_agents. Формат: frontmatter (name, description, tools) + системный промпт с жёсткой привязкой к разделам SPEC и правилам CLAUDE.md. За образец структуры возьми skill-builder. Каждого покажи мне на ревью до сохранения.»

**3. Ре-анализ спеки новой командой:**
> «Два прохода по SPEC.md. Проход 1 — бизнес (red-team §1–5 и §16): где предпосылки слабые, какие цифры могли устареть, что проверяется только руками. Проход 2 — технический (§7–15, §21–23): противоречия, дыры, невыполнимое в статике. Итог — отчёт: критично / желательно / ок + отдельный список того, что могу проверить только я (условия партнёрок, выплаты, домен). Спеку без моего ок не править.»

Gate фазы: правки по отчёту внесены в SPEC (или осознанно отклонены), покрытие фаз агентами 100% → начинаем R1.

Куда класть новых агентов: нужные только этому сайту (например, uk-адаптер) — в `gruzia-site/.claude/agents/`; пригодятся в любом проекте — коммить в свой my_agents, чтобы команда росла вместе с тобой.

Предохранитель: не плоди «супер-агентов» десятками. Агент — это промпт + права на инструменты, а не новые знания: цены прокатов и нюансы границ он не узнает сам, это по-прежнему твои брифы. Каждый лишний агент — токены и координационный шум. Реалистичный дефицит для этого проекта — 3–5 доменных ролей, не двадцать.

## Шаг 6. Рабочий цикл по фазам (что говорить Claude)

Общий принцип: одна фаза = одна-две сессии. Каждую новую сессию начинай с фразы «Прочитай CLAUDE.md и SPEC.md, работаем над фазой R{N}». Всегда ссылайся на номера разделов спеки — это убирает фантазии агентов.

**R0 — утверждение (ты, не Claude).** Закрой §25 спеки: выбери домен (блокирует всё остальное), реши про публичность имени. Впиши решения в SPEC.md, закоммить.

**R1 — каркас.** Промпт:
> «Фаза R1 по SPEC §20 (веди через агента astro-platform-engineer). Создай каркас: Astro 6 + Tailwind 4, структура каталогов из CLAUDE.md, маршруты из SPEC §7, tokens.css с заглушками из §10, команды npm из CLAUDE.md, CI (build + check). Критерий выхода: build и check зелёные, dev-сервер показывает пустые страницы всех разделов.»

После — деплой: на dash.cloudflare.com → Workers & Pages → Create → подключи GitHub-репо `gruzia-site` (Workers Build). Build command `npm run build`, Deploy command `npx wrangler deploy` — каркас уже содержит `wrangler.jsonc` (Workers Static Assets, без SSR-адаптера) и `worker/index.ts` для /go/. С этого момента каждый push виден онлайн (`*.workers.dev`; кастомный домен — в R5).

**R2 — дизайн.** Промпт:
> «Фаза R2. Скиллом ui-ux-pro-max сгенерируй 3 варианта дизайн-системы по брифу SPEC §10 (палитра и шрифты зафиксированы — варьируй остальное). Покажи варианты, жди моего выбора. После выбора зафиксируй токены в tokens.css и сверстай главную по SPEC §8.4.»
Смотри результат на телефоне через превью-URL. Gate: ты принял главную.

**R3 — шаблоны.** Промпт:
> «Фаза R3. Сверстай шаблоны Article/Route/Hub и компоненты из SPEC §9 (включая RouteTimeline и AffiliateBox по §8). Создай схемы коллекций по §11 и одну демо-пару статей ru+uk. Прогони Definition of Done из CLAUDE.md, покажи отчёт по каждому пункту.»

**R4 — конвейер и деньги.** Сначала сам зарегистрируйся: Travelpayouts → Localrent → SafetyWing (проверь способы вывода денег до регистрации). Потом промпт:
> «Фаза R4. Заполни `src/data/partners.json` реальными ID (механизм /go/ — `worker/index.ts` — уже готов с R1, §16). Затем по брифам в briefs/ напиши 3 пары статей по конвейеру из CLAUDE.md. Факты только из брифов, нет факта — TODO.»
Брифы пишешь ты: для каждой статьи — ключевой запрос, 5–7 вопросов читателя, твои факты/цены/фото.

**R5 — запуск.** Подключи кастомный домен в Cloudflare (Workers & Pages → твой проект → Domains), добавь сайт в Google Search Console (подтверждение через DNS), отправь sitemap, включи Cloudflare Web Analytics. Промпт для Claude:
> «Фаза R5, предзапусковый аудит: прогони чек-лист SPEC §21 + security-auditor по §18. Отчёт списком: пункт → статус → что чинить.»

**R6 — рост.** Ритм: 2–3 пары статей в неделю по конвейеру, понедельничный 15-минутный разбор GSC (§17), раз в квартал — аудит репозитория (зависимости/безопасность через агента security-auditor + `npm audit`, дрейф) и обновление денежных статей.

## Шаг 7. Git-гигиена для соло-разработчика

- Работа — в ветках (`r2-design`, `r4-go-redirects`), мерж в `main` после зелёного билда. `main` = то, что в проде.
- Коммиты атомарные, сообщение отвечает «что и зачем». Claude коммитит — ты просматриваешь diff перед push (привычка на 30 секунд, спасает от сюрпризов).
- Перед любым push: `npm run build && npm run check`.

## Шаг 8. Частые проблемы

| Симптом | Причина и лечение |
|---|---|
| `/plugin` не находит marketplace | Старый Claude Code → `claude update`, перезапусти сессию |
| Команды установки не совпадают с этим гайдом | Синтаксис плагинов обновился → README самого репозитория первичен |
| Агенты «придумывают» структуру или факты | В промпте нет ссылок на разделы → всегда «по SPEC §N»; факты — только из briefs/ |
| ui-ux-pro-max предлагает стиль вразрез со спекой | Спека главнее: «в рамках токенов SPEC §10» прямо в промпте |
| Сессия распухла, ответы деградируют | `/compact`, либо новая сессия + «прочитай CLAUDE.md и SPEC.md, фаза R{N}» |
| Билд падает на Cloudflare, локально ок | Сверь версию Node в настройках проекта (переменная NODE_VERSION=20) |
| Дубли скиллов/агентов | Ставил и способ А, и способ Б → удали один (через `/plugin` или почисти `.claude/` проекта) |

## Шаг 9. Чего НЕ делать

- Не клонировать my_agents и ui-ux-pro-max внутрь `gruzia-site` и не добавлять их как git submodules.
- Не начинать вёрстку до утверждения токенов (R2) и не начинать массовый контент до рабочего /go/ (R4).
- Не подключать все 33 плагина ruflo сразу — только базу из Шага 3, остальное по фазам.
- Не пропускать gates: «почти готово» в фазе N всегда дороже чинить в фазе N+2.

## Что дальше (первые три действия прямо сейчас)

1. Выполни Шаги 1–2 (окружение + репозиторий со SPEC.md и CLAUDE.md).
2. Выполни Шаги 3–5 (подключение обоих инструментов + контрольная проверка).
3. Закрой R0 (выбери домен), прогони R0.5 из Шага 5.5 — и запускай R1 промптом из Шага 6.
