# CONTENT_GUIDE.md — как добавить SEO-статью на Montenegro Guidebook

Рабочая инструкция для регулярного написания статей. **Источники правды по продукту/правилам:** `SPEC.md` → `CLAUDE.md` → `docs/STRATEGY.md` → этот файл. Если что-то противоречит — приоритет у SPEC/CLAUDE/STRATEGY.

> TL;DR: статья = Markdown-файл в `src/content/articles/<lang>/<slug>.md` (Astro Content Collections). **EN — ведущий язык** (на корне `/`), пишется первым; RU → `/ru/…`. **Языки сайта: en + ru (uk не используется, удалён 2026-06-22). Публикуется парой en + ru** одним коммитом (gate `check-parity` требует пару en+ru), иначе не пройдёт. Перед публикацией — `npm run qa` → должно быть **GO**.

---

## 0. Для владельца: как это работает (простыми словами)

**Главное: статьи пишешь НЕ ты.** Их пишу я: черновик **en** (ведущий) → перевод ru → фото → **взаимная перелинковка** (новая статья ссылается на соседей+хаб — это гейт; И опорная/соседи кластера ссылаются на неё, в обоих языках — мой ручной шаг, гейтом не ловится) → проверка фактов → `npm run qa` (= GO) → коммит + публикация. Твоя роль короткая:

1. **Дать факты, которых нет в открытых источниках** — особенно местное по побережью (Будва/Котор/Тиват: любимые места, цены по чеку, что реально стоит/не стоит, что закрылось). Без этого я пишу только на **проверяемых фактах** из надёжных источников — без выдуманного «я там был». _(Источник живого опыта — открытая развилка владельца, `docs/STRATEGY.md` §5.)_
2. **Дать свои фото** (по желанию) — они дают метку «Проверено» и сильнее в Google, чем сток. Нет своих → беру бесплатные CC.
3. **Сказать «ок»** на публикацию.

**Как заказать статью:** просто напиши «пиши про &lt;тему&gt;» — я беру тему из плана, собираю факты, пишу пару en/ru, прогоняю гейты и публикую. Или кинь мне факт/место — я превращу в контент.

**По какому плану:** стратегия и приоритеты — `docs/STRATEGY.md` (EN-first/Tier-1, порядок кластеров), рыночная фактура — `docs/MONTENEGRO-ANALYSIS-2026-06-20.md`. Детальный кластерный план с целевыми EN-запросами — анализ §5; рабочая очередь `docs/CONTENT_PLAN.md` (заводится в Фазе 3).

**Что пишем (порядок приоритета кластеров, EN-first):**
1. **Планирование** (верх воронки: montenegro travel guide, best time to visit, getting around) — ведёт трафик вниз.
2. **MONEY: аренда авто / страховка / eSIM** (car rental in montenegro, montenegro travel insurance, best esim) — транзакционные, AIO-стойкие.
3. **Релокация/номады** (EN-first: cost of living, moving to montenegro, register company, taxes, nomad visa) — дороже, менее сезонно, рекуррент.
4. **Достопримечательности / Города / Маршруты** (things to do, Kotor/Budva guides, road trip itinerary).
5. **Еда** (what to eat, where to eat in Kotor/Budva).
6. **Развлечения / события** (beaches, Sea Dance / KotorArt — афиша как SEO-магнит).

**Правило фактов (железное, §15):** цены/адреса/часы — только из надёжного источника + «уточняйте» + дата; нет данных → не выдумываем. Валюта — **евро (€)**. YMYL (визы/налоги/ВНЖ/страховка) — только официальные источники (gov.me/MUP/PwC) + дисклеймер. Это защита от бана Google.

## 1. Как технически устроены статьи

- **Движок:** Astro 6 + **Content Collections**. Статьи — это **Markdown** (`.md`/`.mdx`) с YAML-frontmatter. Отдельные `.astro`-файлы под каждую статью НЕ создаются — рендерит общий шаблон `src/layouts/ArticlePage.astro` через динамический маршрут.
- **Где лежат файлы:**
  ```
  src/content/articles/en/<slug>.md   ← английская версия (ведущая)
  src/content/articles/ru/<slug>.md   ← русская (обязательна, см. §3)
  ```
  (Маршруты — отдельная коллекция `src/content/routes/`; «Где поесть» — `src/content/restaurants/`; услуги — `src/content/services/`; города-карточки — `src/content/cities/`. Эта инструкция — про **статьи** в `articles/`.)
- **Как файл попадает в нужный раздел:** разделом управляет поле **`category`** во frontmatter, а НЕ папка. URL строится как `/<lang-префикс>/<category>/<slug>/`. Файл `articles/en/abc.md` с `category: 'cities'` → страница `/cities/abc/` и автоматически попадает в хаб `/cities/` (хаб сам подтягивает статьи своей категории — ручных списков нет).
- **Допустимые `category`** (enum в `src/content.config.ts`):
  `attractions` · `cities` · `food` · `entertainment` · `routes` · `transport` · `car-rental` · `relocation` · `insurance` · `news` · `planning`.
  - Для обычных SEO-статей чаще всего: **planning, cities, attractions, transport, car-rental, relocation, news, entertainment**.
  - Нюансы: `routes` — это отдельная коллекция `routes/` (не `articles/`); `food` — директория мест (`restaurants/`), а не лонгрид; `insurance` — раздел с кастомным хабом, но статьи категории `insurance` в `articles/` тоже работают (`/insurance/<slug>/`).

## 2. Frontmatter статьи (поля)

Полная схема — `src/content.config.ts` (`articleBase`). Главное:

| Поле | Обяз.? | Что это |
|---|---|---|
| `title` | да | Заголовок = **H1** на странице. **≤ 60 символов** (гейт `check-parity`). Должен содержать целевой запрос (см. §8). |
| `description` | да | Meta description. **≤ 155 символов** (валидируется схемой). 1–2 предложения с запросом. |
| `slug` | да | Латиница-цифры-дефис. **Стабилен навсегда** (URL не меняем — правило 3). Одинаковый во всех языках. |
| `lang` | да | `'en'` / `'ru'` — должен совпадать с папкой. |
| `category` | да | Один из enum выше → определяет раздел и URL. |
| `publishedAt` | да | Дата публикации `YYYY-MM-DD`. |
| `updatedAt` | да | Дата обновления `YYYY-MM-DD` (видна на странице — «Обновлено · …»). Поднимай при правках фактов. |
| `cover` | да* | Обложка: `src` (путь к webp) + `alt`. *Фото обязательны (правило проекта); гейт `check-photos` валит сборку без существующего cover ≤200 КБ. |
| `coverCredit` | для CC | Атрибуция обложки «Фото: Автор / источник, лицензия» — обязательна для CC/чужих фото (§18). |
| `gallery` | нет | Массив `{src, alt, credit?}` — мини-галерея с лайтбоксом. |
| `affiliate` | нет | 1–3 партнёрских бокса (см. §7). По умолчанию `[]`. |
| `sources` | нет | Массив доменов/ссылок-первоисточников фактов (§14). |
| `verifiedAt` | нет | Дата реальной сверки места → бейдж «Проверено · …». Без сверки — НЕ ставить (будет «Обновлено», правило 4). |
| `geo` | нет | `{ coord: [lat, lng], address? }` — для статей о местах (даёт карту + schema). |
| `accessFrom` | нет | «Как доехать» из Подгорицы/Тивата/Будвы (`{km?, duration?, note?}`). |
| `visit` | нет | «Вход и часы»: `{price?, hours?, note?, checkedAt?}`. Валюта — €. Цены/часы НЕ выдумывать — только источник + дата + «уточняйте». |
| `featuredOrder` | нет | Меньше число = выше в хабе раздела. |
| `attractionType` / `region` | нет | Только для `attractions` (фильтры каталога). Слаги — в `content.config.ts` (`region` = `coastal`/`central`/`northern`). |
| `razvlType` | нет | Только для `entertainment` (`kazino`/`mesta`/`nochnaya-zhizn`/`afisha`/`aktivnyy`). |
| `tags` | нет | Свободные теги (показ в смежных разделах). |
| `hotelWidget` | нет | `true` → виджет отелей Trip.com (для городов с высоким спросом на жильё). |
| `draft` | да на старте | `true` = скелет, в сборку НЕ попадает. Снять (`false`) только когда статья готова и гейты зелёные. |

**Примеры (плейсхолдеры — будущие черногорские статьи; смотри похожие после наполнения):**
- `planning/montenegro-travel-guide` — справочная статья верха воронки: cover + внутренние ссылки вниз (маршруты/города/аренда/страховка).
- `cities/kotor` — город: `featuredOrder`, `hotelWidget`, `visit`, geo.
- `attractions/durmitor` — статья о месте: `geo`, `gallery`, инлайн-фигуры, `accessFrom`, `region: 'northern'`.
- `insurance/montenegro-travel-insurance` — денежная статья: cover + `affiliate` (SafetyWing для EN / EKTA для ru) + `sources`.

## 3. Мультиязычность (EN/RU) и hreflang

- **EN — ведущий язык по умолчанию, на корне `/`.** RU → `/ru/…` (x-default → en). uk не используется (удалён 2026-06-22, старые /uk/-URL → 301 на корень). Префикс задаёт `langPrefix()` в `src/i18n/index.ts` (`DEFAULT_LANG='en'`). **EN пишется первым** и задаёт целевой запрос/приоритет (EN-first, `docs/STRATEGY.md`).
- **Чтобы EN-статья попала на корень:** файл в `src/content/articles/en/<slug>.md`, `lang: 'en'`, `category: '<…>'`. URL получится `/<category>/<slug>/` (маршрут `src/pages/[category]/[slug].astro`). RU — аналогично в своей папке с тем же `slug`.
- **🔴 Главное правило (CLAUDE правило 1):** материал = **en (ведущий) + ru** с **одинаковым `slug`**, публикуется парой одним коммитом. Гейт `check-parity` требует пару **en + ru** (половину пары не публиковать). ru — крепкий вторичный рынок, но приоритет и подбор тем — от EN-интента.
- **hreflang — автоматический.** `BaseLayout` сам проставляет взаимные `<link rel="alternate" hreflang="…">` + `x-default`→en по тем языкам, где **существует тот же `slug`**. От тебя нужно только: одинаковый slug в en/ru — и связка построится сама. Переключатель языка тоже работает по совпадающему slug (`mirrorPath`).
- **Перевод ru:** адаптируй под аудиторию (а не дословно): для CIS — EKTA вместо недоступных SafetyWing-страховок, въезд/логистика из СНГ, валютные нюансы (€). Числовые поля (даты, цифры) и `slug`, `cover.src`, `gallery` src — одинаковые в паре.

## Как делать перевод RU правильно

RU — обязательный перевод ведущей EN-версии (правило 1) и **полноценная статья**, а не технический довесок:

1. **Живой язык — не калька.** Текст должен читаться как написанный носителем: естественные конструкции, лексика и идиомы. НЕ пословный перевод английского — переводим смысл и тон, а не слова.
2. **Факты идентичны EN.** Цены, цифры, расстояния, даты, названия — те же, что в EN-версии (как и `slug`, `cover.src`, `gallery` src, числовые поля frontmatter). Расхождение фактов между версиями недопустимо.
3. **H1 = локальный целевой запрос.** `title` ru версии содержит то, **как ищут носители языка в Google** (русский поисковый запрос), а НЕ перевод английского H1. При этом `title` **≤ 60 символов**: длинный запрос → ядро в H1, хвост в H2/H3. Аналогично — `description` (≤155) под локальный запрос.
4. **Внутренние ссылки — на нужный язык.** В RU-версии все внутренние ссылки с префиксом `/ru/…`, в EN — без префикса (`/…`). Минимум 2.
5. **Тело — локальные H2/H3** под смежные запросы, своя лид-фраза, подписи к фото на языке версии. Кредиты фото после «Фото:» (автор/лицензия) НЕ переводим — оставляем дословно.

## 4. Пошагово: добавить статью (EN-first)

1. **Бриф/факты.** Собери факты заранее (цены €, адреса, часы — из надёжных источников). Нет факта → ставь `<!-- TODO -->` и не выдумывай (правило 4). Можно завести `briefs/<slug>.md`.
2. **Скелет (рекомендуется).** Из корня проекта:
   ```
   npm run new -- article <slug> --category <category> --title "Title ≤60 chars"
   ```
   Создаст `articles/{en,ru}/<slug>.md` (схема-корректный frontmatter, `draft: true`, DoD-чеклист в комментарии) + папку фото `public/images/<slug>/`. Существующие файлы не перезатирает.
   *(Альтернатива — скопировать существующую статью и поменять frontmatter.)*
3. **Текст EN (ведущий).** Заполни `articles/en/<slug>.md`:
   - `description` (≤155), `publishedAt`/`updatedAt`.
   - **Тело:** начинается с лид-абзаца (первый абзац, без `#`), затем секции `## H2` / `### H3`. **`#`/H1 в теле НЕ ставить** — H1 рендерится из `title`.
   - Соблюдай SEO-требования из §8.
4. **Фото** (§6) — cover обязателен + 2–4 инлайн-фигуры или галерея.
5. **Внутренние ссылки** (§7) — минимум 2 на смежные статьи + ссылка на хаб раздела.
6. **Перевод RU** — создай/заполни `articles/ru/<slug>.md` (тот же slug).
7. **Сними `draft`** → `draft: false` в обеих версиях.
8. **Гейты** (§9) — `npm run qa` → **GO**.
9. **Коммит пары** en+ru одним коммитом → push в `main` (Cloudflare соберёт и задеплоит).

## 6. Фотографии

- **Уникальные на статью (правило владельца):** у каждой статьи СВОИ фото — **НЕ переиспользовать кадры из других статей** (`/images/<другой-slug>...`). Под новую статью искать новые кадры через пайплайн и класть в свою папку. Проверять, что НА фото (не выдавать чужую страну/валюту за черногорскую). Память `article-photos-unique`.
- **Где лежат:** `public/images/<slug>/` (создаётся скаффолдером). Формат — **`.webp`**, каждое **≤ 200 КБ** (перф-бюджет §15), с заданными `width`/`height` (нулевой CLS).
- **Откуда брать:** только свои фото или **CC/PD** (Wikimedia / Openverse / Unsplash). Скрипты-помощники: `scripts/commons-candidates.mjs`, `scripts/openverse-candidates.mjs`, `scripts/unsplash-candidates.mjs` (поиск кандидатов) → `scripts/build-gallery.mjs` / `build-unsplash.mjs` (скачать + сжать в webp ≤200 КБ). Атрибуцию (автор+лицензия) бери **дословно из источника**, не из имени файла.
- **Обложка (frontmatter):**
  ```yaml
  cover:
    src: '/images/<slug>/cover.webp'
    alt: 'Meaningful description of the shot'
  coverCredit: 'Photo: Author Name / Wikimedia Commons, CC BY-SA 4.0'
  ```
- **Инлайн-фото в тексте** (ключевые кадры — прямо в теле, с подписью). Точная разметка (raw HTML в .md):
  ```html
  <figure class="figure">
  <img src="/images/<slug>/g1.webp" alt="What is in the photo" width="1280" height="853" loading="lazy" decoding="async" />
  <figcaption>Caption — what is in the photo. <span class="figure__credit">Photo: Author / source, license</span></figcaption>
  </figure>
  ```
  `width`/`height` = реальные пиксели файла (иначе перекос/CLS).
- **Мини-галерея внизу** (необязательные кадры) — через frontmatter:
  ```yaml
  gallery:
    - src: '/images/<slug>/g2.webp'
      alt: '…'
      credit: 'Photo: …'
  ```
  Каждое фото используется **один раз** (либо инлайн, либо в галерее).
- **`alt`** — осмысленный, по делу (не «photo1»); описывает содержимое кадра.
- **После добавления обложек** прогони `npm run build:covers` — сгенерит адаптивные варианты (`srcset`) и обновит манифест `src/data/cover-variants.json` (мобайл грузит меньший размер). Варианты коммитятся.

## 7. Внутренние ссылки

- **Формат:** обычная Markdown-ссылка с **префиксом языка** и завершающим слэшем. EN (корень) — без префикса, RU — `/ru/…`:
  ```markdown
  See the [Kotor guide](/cities/kotor/) and the [car rental](/car-rental/) section.
  ```
- **Минимум 2** внутренние ссылки в теле на смежные статьи — это **гейт** `check-interlinks` (DoD §8.1). Партнёрские `/go/`-ссылки и якоря в счёт не идут.
- Ставь ссылку и **на хаб раздела** (`/<category>/`), и желательно встречную ссылку из смежной статьи (двунаправленный кластер).
- **Партнёрские ссылки — ТОЛЬКО через `/go/`** (правило 2), и только во frontmatter `affiliate` (рендерится как AffiliateBox), не прямыми URL в тексте:
  ```yaml
  affiliate:
    - partner: 'trip-hotels'      # ключ из src/data/partners.json
      label: 'Where to stay'
      position: 'body-end'
  ```
  Сегментация по языку (SPEC §16): EN → SafetyWing/DiscoverCars/GetYourGuide/Booking; CIS (ru) → EKTA/Localrent/Aviasales.

## 8. SEO-требования к статье (соблюдаем ВСЕГДА)

1. **H1 = целевой запрос.** Поле `title` (= H1) точно содержит целевой поисковый запрос (EN-first — целимся в EN-интент). Ограничение: **≤ 60 символов** (гейт). Один H1 на странице (он из `title`; в теле `#` не использовать).
2. **Первый абзац — прямой ответ на запрос в 2–3 предложениях.** Лид сразу даёт суть, без «разгона».
3. **Подзаголовки `## H2` / `### H3` со смежными запросами** — структура под вопросы читателя (что/где/как/сколько/когда). Логичная иерархия без скачков уровней.
4. **Таблицы** для цен / расстояний / сравнений — обычные Markdown-таблицы (`| … | … |`). Числа — только из источника (правило 4), цены (€) — с «уточняйте» + дата.
5. **2–4 фото** с осмысленными `alt` и подписями (figcaption) — см. §6.
6. **Блок внутренних ссылок** на смежные статьи — минимум 2 (см. §7), плюс хаб раздела.
7. **Дата обновления** — поле `updatedAt` (видно на странице). Поднимать при правке фактов.
8. **Объём 1200–2000 слов, без «воды».** Каждый абзац несёт пользу; никакой генерёнки и near-duplicate (§15 — риск бана).
9. **Доп. требования проекта:** `description` ≤155 с запросом; факты не выдумывать (правило 4) — нет данных → TODO/«уточняйте» + источник в `sources`; YMYL → официальные источники + дата; уникальная фактура (≥3 своих факта), не перепечатка; **FAQ не используем** (убран со всего сайта — новую инфо вплетать в тело).

## 9. Проверка перед публикацией (гейты)

Из корня проекта:

| Команда | Что проверяет |
|---|---|
| `npm run check` | типы + соответствие frontmatter схеме |
| `npm test` | enums · **parity (есть ли пара en+ru, slug, title ≤60)** · photos (cover ≤200 КБ) · **interlinks (≥2)** |
| `npm run test:links` | внутренние ссылки резолвятся + `/go/`-партнёры валидны |
| `npm run lint` | стиль/ESLint |
| **`npm run qa`** | **всё вышеперечисленное + чистая сборка + аудит `dist/` → вердикт GO/NO-GO** |

**Правило:** перед коммитом — `npm run qa` = **GO**. Затем коммит пары en+ru + push в `main`.

**Как убедиться, что статья на корне с корректным hreflang:**
1. `npm run build` → проверь, что появился `dist/<category>/<slug>/index.html` (EN) + `dist/ru/…`.
2. В нём есть `<link rel="alternate" hreflang="en" …>`, `hreflang="ru" …>` + `x-default`→en — проставляется автоматически по совпадающему slug.
3. `npm run test:links` — без битых ссылок.

## 12. Как добавлять ЗАВЕДЕНИЕ (ресторан / услугу) — это делаю я

Заведения — это НЕ лонгрид-статьи, а карточки-директории:
- **Ресторан / кафе / бар / винотека** → коллекция `restaurants/` → раздел «Где поесть» `/food/`.
- **Услуга** (риелтор / клининг / переезд / ремонт / документы / переводчики) → коллекция `services/` → `/relocation/services/`.

### Жёсткие правила отбора (обязательно)
1. **Только реальные заведения** с проверяемыми данными. Вымышленные/«примеры» в директорию проверенных мест НЕ вносим (правило 4, §15).
2. **Консенсус > 4★ из НЕСКОЛЬКИХ источников** + наш `ratingNote`/`review` своими словами (чужие отзывы дословно нельзя, §15).
3. **Фото обязательно** (`cover`; для CC — `coverCredit`). Цены (€)/часы (`dishes`/`hours`) — только из источника, иначе не указывать.
4. **Пара en + ru** (en — ведущий), одинаковый `slug`.
5. **🔵 Метка «Проверено» — ОБЯЗАТЕЛЬНА** (правило владельца, §13 ниже).

### Шаги
1. Скаффолд: `npm run new -- restaurant <slug> --city "Budva"` (или `service <slug>`) → создаст en/ru + папку фото.
2. Заполни обязательные поля. **Ресторан:** `name`, `city`, `priceLevel` (€/€€/€€€), `geo.coord`, `mapUrl`, `ratingNote`, `sources`, `summary`, **`verifiedAt`**, `cover`. **Услуга:** `name`, `rubric`, `summary`, реальный `url`/`contact` (без example.com), **`verifiedAt`**.
3. Фото (§6); `verifiedAt` = дата нашей **реальной** сверки данных.
4. Перевод ru → снять `draft` → `npm run qa` = GO → коммит.

### Добавить заведение в витрину на главной (с флагом города)
В `HomePage.astro` → `showcasePicks` добавить pick с полем `city` (`city: 'Budva'`) → на карточке появится пин-чип «📍 Город» (см. память `home-showcase-monetization`). В витрину помещаем только заведения с меткой «Проверено» (§13). Города директории еды — Budva/Kotor/Podgorica (`where-to-eat-<город>`).

## 13. 🔵 Правило: метка «Проверено» обязательна у КАЖДОГО заведения/услуги

**КАЖДОЕ заведение и услуга, которые мы вносим, обязаны нести видимую метку «Проверено» (галочка ✓)** — это обещание директории «проверенные места» и ключевой сигнал доверия (E-E-A-T).

- **Чем:** единый компонент **`VerifiedBadge`** (галочка ✓ + «Проверено · {дата}» + расшифровка «опытными путешественниками, гидами и местными»). Дата — из поля `verifiedAt`. **Не изобретать новых галочек** — один знак доверия на весь сайт (консистентность).
- **Честность (правило 4):** `verifiedAt` ставим ТОЛЬКО при реальной сверке (данные сверены с источниками/на месте). «Проверено» на непроверенном — запрещено.
- **Текущее состояние:** рестораны (`/food/`) — `verifiedAt` обязателен схемой → бейдж «Проверено». Услуги (`/services/`) — поле `verifiedAt` в схеме `services` + компактный `VerifiedBadge` (✓ + «Проверено», без даты) в карточках; ставить только реально сверенным (демо/вымышленные — без).
- **Компонент:** `VerifiedBadge` имеет `compact` (для плотных карточек — ✓ + «Проверено») и полный (✓ + «Проверено · {дата}» + расшифровка — для страницы места/шапки статьи).
- **Метка «Проверено» — БЕЗ даты у ВСЕХ заведений** (решение владельца; даты не добавлять): `RestaurantCard` всегда рендерит компактный бейдж без даты. `verifiedAt` остаётся обязательным в схеме (хранится), но в карточке не показывается.
- **Знак качества** — золотая медаль `/images/quality/seal-gold.webp` (запасные SVG-печати `seal-star`/`seal-rosette`/`seal-shield`/`seal-laurel`/`seal-check` там же).
  - **ОПТ-ИН, только по запросу владельца.** Поле `qualityMark: true` у ресторанов И у статей (`articleBase`); по умолчанию выкл. **Перед созданием любого заведения — СПРОСИТЬ владельца, нужен ли знак.**
  - **Размер ФИКСИРОВАН и един:** карточки (витрина `.scard__quality`, `/food/` `.rcard__quality`, миниатюра хаба `.card__quality`) — **62px**, выходит за правый верхний угол + золотое свечение; обложка-герой статьи (`.cover__quality`) — 72px. Не делать адаптивным.
  - Сменить знак везде = поменять путь в `ShowcaseRail.astro` (`QUALITY_MARK`), `RestaurantCard.astro`/`ArticleCard.astro`/`ArticlePage.astro` (`*__quality` src). Память `quality-mark-establishments`.
- **Сайт + Instagram заведения** (правило владельца): **отдельные поля** `website` (личный сайт → кнопка «Сайт», `globe`) и `instagram` (→ кнопка «Instagram») — у ресторанов и статей-мест. **При добавлении ЛЮБОГО заведения СПРАШИВАТЬ у владельца Instagram** (память `establishment-instagram`). **Личный сайт НЕ убирать** — показываем обе кнопки, если есть и сайт, и IG.
- **Фото заведений: карусель + лайтбокс.** Несколько фото = `cover` + `gallery[]`; в карточке листаются (`/js/rcard-gallery.js`), клик → полноэкранный просмотр (`/js/lightbox.js`).
- **Витрина на главной поддерживает заведения:** в `HomePage.astro` → `showcasePicks` добавить `{ kind: 'restaurant', slug: '<slug>', edaSlug: 'where-to-eat-<город>', kicker: 'food' }` → карточка с обложкой, ссылкой на страницу города в «Еде» и пин-чипом города. В витрину — только заведения с меткой «Проверено».
